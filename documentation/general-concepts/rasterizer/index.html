Rendering ink content

The Rasterizer module provides GPU accelerated stroke-rendering functionality by utilizing various drawing back-ends.

As of the current version, WILL implements four rendering back-ends:

  • DirectX 11.1 for Windows
  • OpenGL ES 2.0 with EGL for Android
  • OpenGL ES 2.0 with EAGL for iOS
  • WebGL for Web

The Rasterizer interface uses a similar level of abstraction to other popular 2D graphic library interfaces but is specialized for rendering strokes. The Rasterizer module also provides mechanisms for advanced scene composition techniques, such as virtual (off-screen) layers, masking, blending, and transformation.

Because the Rasterizer module is highly specialized, it depends on interoperability with traditional 2D graphics libraries to provide the richness of features required for digital ink use cases, such as the combination of multiple media (text, images, and so on) with ink content.

Conceptual API design

In general, the 2D graphics libraries of the supported platforms have a similar design on a conceptual level. From this perspective, the purpose of the Rasterizer module is as follows:

  1. Introduce digital ink rendering on a conceptual level.
  2. Provide a platform-specific API that follows each platform's 2D graphics library paradigms and adds digital ink rendering functionality.

Drawing geometric primitives

On a conceptual level, the Rasterizer module provides an API for drawing geometric primitives described by a stroke mathematical model. A geometric primitive is a path containing a plurality of control points that are interpolated using a Catmull-Rom spline. The drawing algorithm creates strokes with variable stroke width and stroke opacity.

Interoperability

The Rasterizer module provides an API for interoperability with a platform’s 2D/3D graphics libraries. The API provides methods for including digital ink content within a 2D/3D scene created with a platform’s graphics libraries, and for including 2D/3D elements created with a platform's graphics libraries in a scene created by WILL. The functionality and performance of these methods depend on platform-specific interoperability mechanisms. However, this API is essential to the functions of the Rasterizer module, so its methods are implemented in the way that performs best.

2D graphics operations

The Rasterizer module also provides a complementary utility API to perform traditional 2D graphics operations like the following:

  • Managing multiple layers
  • Specifying clipping rectangles
  • Filling paths with a solid color (for more information, see the tutorials on masking images for the desired platform)

This complementary utility API is included for performance reasons. These operations might perform better than the corresponding operations from the platform’s 2D graphics library (invoked through the platform’s interoperability mechanisms). However, this depends on the hosting platform.

The preliminary path technique for reducing perceived lag is a good example of a performance-sensitive feature: one way to implement this technique requires layer management with minimum performance overhead. In this case, the complementary utility API for managing multiple layers in the Rasterizer module is ideal. For more information, see the tutorials on preliminary paths for the desired platform.

Rasterization techniques

The geometric primitives specified with the stroke mathematical model are paths containing control points that are interpolated using a Catmull-Rom spline. In addition to the control points, the stroke mathematical model specifies the values of the interpolation parameter in the first and last segments, in which the stroke starts and ends respectively. If required, all paths can be created with variable width and opacity. The rasterization techniques that are implemented in the Rasterizer module are techniques for stroking paths that have these characteristics.

Shape-filling technique

This is a solid color technique for rasterizing strokes that have variable width. This method is more limited in terms of expressiveness, but performs better than more complex rendering techniques.

This technique fills the stroke boundary of a path with solid color. Because the implementation of the stroke boundary calculation is designed for optimal performance, you can use the shape-filling technique only to rasterize strokes that have constant opacity. The normal method of rasterizing a stroke with constant opacity uses an additional layer and a specific blend mode. However, in some simpler cases this technique can be configured to rasterize a stroke that has constant opacity without using a separate layer, and to utilize more blend modes. For this reason, you can specify the shape-filling technique with two brush types: SolidColorBrush and DirectBrush. The SolidColorBrush is the usual and recommended way to use this technique, but the DirectBrush can be suitable in some cases (for example, when you can’t afford to use additional layers).

SolidColorBrush usage in combination with multiple layers is simplified by the StrokeRenderer component, which is a platform-specific component that is available for all supported platforms. For more information, see the tutorials on building paths for the desired platform.

The shape-filling technique is suitable for typical scalable vector graphics usages. Because of its simplicity in terms of expressiveness, strokes that are rasterized with this technique can be easily manipulated (split), scaled, redrawn, and so on.

Particle-scattering technique

This method renders strokes using overlapping particles. This technique allows you to build more expressive tools (such as crayon, pencil, or watercolor brushes) by controlling the following parameters:

  • The shape parameter specifies the shape of the particle. It accepts an image in which the non-transparent pixels define the particle shape. You can also pass a sequence of images to this parameter. In this case, the method interprets them as mipmaps.
  • The spacing parameter specifies the distance between different particles along the path trajectory.
  • The scattering parameter specifies the maximum offset in a direction that is perpendicular to the path trajectory in which the particle is positioned. The actual position of the particle is randomly determined within the interval defined by this parameter.
  • The rotation parameter specifies the rotation of the particle. It accepts three possible values:
    • none indicates that the shape will not be rotated.
    • random indicates that the shape will be rotated randomly before it renders.
    • trajectory indicates that the shape will be rotated to match the path trajectory.
  • The fill parameter specifies the fill pattern of the shape. It accepts an image from which the pattern is sampled. You can also pass a sequence of images in this parameter. In this case, the method interprets them as mipmaps.
  • The fillSize parameter specifies the size of the fill image from which the pattern is sampled. It accepts 2D values and can be implemented with two separate parameters: width and height.
  • The randomizeFill parameter specifies whether the fill sampling location is to change randomly for each particle rendering.

The particle-scattering technique supports both constant and variable opacity and all blend modes. This makes it the preferred choice for raster-based applications. However, this technique does not support typical vector graphic manipulations, such as splitting.

Interoperability with platform-specific graphics libraries

The interoperability methods provided in the Rasterizer module can be divided into two types:

  • Interoperability with platform 2D graphics library
  • Interoperability with platform 3D graphics library

Because the Rasterizer module implementation is based on platform-specific 3D graphics libraries, the second type of interoperability is more straightforward. The first type of interoperability is limited by the platform interoperability mechanisms between 2D and 3D graphics libraries.

In general, the Rasterizer module is designed to reuse and integrate as much as possible from 2D graphics paradigms. It is also designed to expose all necessary parameters to allow full interoperability with 3D graphics libraries.
